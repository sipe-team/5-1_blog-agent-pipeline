# 이번 주 진행 내용

## 1. Google Calendar 연동

### 추가된 기능

자연어 메시지에서 일정을 추출하여 Google Calendar에 자동 등록.

```
사용자 메시지 → Haiku (날짜/제목 추출) → Google Calendar 등록 → Slack 알림
```

### 기술 스택

- Google Calendar API (OAuth 2.0)
- Haiku: 날짜/시간/제목 구조화 추출

### 문제해결: 중복 이벤트 등록 방지

Slack은 Lambda가 3초 내에 응답하지 않으면 동일 이벤트를 재전송한다. Lambda 콜드 스타트로 처리가 3초를 초과하는 경우가 발생했고, 일부 재시도는 `X-Slack-Retry-Num` 헤더 없이 도달하여 필터링이 불가능했다. 결과적으로 캘린더에 동일 일정이 중복 등록되는 문제가 발생.

**결론:**
Slack `message_ts`를 Google Calendar 이벤트 ID로 사용. 동일 메시지는 항상 동일한 이벤트 ID를 생성하므로 아무리 재시도가 와도 중복 삽입이 원천 차단된다. Google Calendar 409 응답 시 기존 이벤트를 반환하고 Slack 알림은 재전송하지 않는다.



## 2. 일정 조회/수정/삭제

### 추가된 기능

- 자연어로 일정 조회, 수정, 삭제
- 복수 후보 매칭 시 사용자에게 번호 선택 요청 후 처리

**흐름 (수정/삭제 예시):**

```
"금요일 점심 약속 수정해줘"
  → Haiku: 날짜 범위 추출
  → Google Calendar: 해당 기간 전체 조회
  → Haiku: 목록 + 요청 비교 → 후보 선택
    확신 → 바로 처리
    불확실 → "1번이에요, 2번이에요?" 번호 선택 요청
    없음 → "일정이 없어요" 안내
```

**복수 후보 처리 흐름:**

```
후보 질문 → DynamoDB에 대기 상태 저장 (5분 TTL)
다음 메시지 "2번이요" → 대기 상태 조회 → 바로 처리
```

### 기술 스택

- DynamoDB: 무상태(stateless) Lambda 간 대기 상태 저장 (TTL 5분)
- Haiku: 맥락 기반 이벤트 매칭

### 문제해결: 이벤트 검색 방식 선택

처음에는 사용자 메시지에서 키워드를 추출해 캘린더 이벤트 제목과 문자열 매칭하는 방식을 구현했다. 그런데 "점심 약속 수정해줘"에서 추출한 키워드 "점심"이 실제 이벤트 제목 "현영 선배님, 영빈"과 매칭되지 않아 검색이 실패했다. 한국어 자연어의 특성상 사용자가 표현하는 방식과 실제 저장된 이벤트 제목이 다른 경우가 많았다.

TF-IDF, 한국어 특화 임베딩 벡터 방식도 검토했으나, 별도 모델 필요 + 추가 비용 + 구현 복잡도가 개인 프로젝트 규모에 맞지 않는다고 판단.

**결론:**
이미 사용 중인 Haiku에게 맥락 매칭을 위임. 날짜 범위 내 전체 일정을 가져온 뒤 Haiku가 사용자 요청과 이벤트 목록을 비교해 가장 유사한 것을 선택하게 했다. 추가 비용 최소화 + 한국어 맥락 이해력이 문자열 매칭보다 월등히 높다는 두 가지 이점을 동시에 확보.

### 문제해결: 입력 의도 불일치 문제

"내일 현영 선배님, 영빈과 점심" 메시지에서 AI가 "점심"을 이벤트 제목으로, "현영 선배님, 영빈"을 참석자로 해석했으나 사용자 의도는 제목이 "현영 선배님, 영빈"이었다. 자연어는 근본적으로 모호성이 있다.

세 가지 해결책을 검토:

- **A. 등록 전 확인 단계 추가** → 응답 왕복이 2회로 늘어 UX 저하
- **B. 슬래시(/) 구분자 형식 제공** → 즉시 적용 가능, 학습 비용 낮음
- **C. Slack Block Kit Modal** → 가장 명확하지만 구현 복잡, 장기 과제

**결론:**
B(슬래시 구분자) 채택 + 수정 기능으로 사후 보정. `날짜시간 / 제목 / 장소` 형식으로 입력하면 AI가 모호하게 해석할 여지를 제거. 빠른 적용과 사용성을 모두 확보.



## 3. Notion 연동 — 인사이트 & 일기

### 추가된 기능

- 인사이트: URL 스크래핑 → Notion DB 자동 저장 (제목, 본문, 태그 자동 분류)
- 일기: 하루치 메시지 버퍼링 → 새벽 4시 AI 재작성 → Notion DB 발행

**일기 흐름:**

```
메시지 수신 → DynamoDB 버퍼 저장
새벽 4시 (EventBridge cron)
  → 버퍼 전체 수집
  → Sonnet: 일기 문체로 재작성 + 태그 분류
  → Notion DB에 날짜·태그 포함하여 발행
```

**일기 날짜 경계:** 04:00 KST ~ 익일 03:59 KST

- 자정을 넘겨도 자기 전 기록은 심리적으로 당일 경험이므로, 04:00을 하루 시작점으로 정의.

### 기술 스택

- DynamoDB: 일기 버퍼 (`diary_date` + `message_ts` 복합 키)
- AWS EventBridge: cron 트리거 (`04:00 KST` 일기 발행 / `06:00 KST` 모닝 브리핑)
- Sonnet: 일기 재작성 (품질 우선, 배치 처리로 속도 무관)
- Haiku: 태그 분류 (단순 분류 태스크, 비용 절감)

### 문제해결: 태그 자동 분류 일관성

**인과관계:**
AI가 매번 새로운 태그를 자유 생성하면 Notion DB의 태그가 파편화되어 필터링이 불가능해진다. 그렇다고 태그를 고정 목록으로만 제한하면 새로운 주제에 대응하지 못한다.

**결론:**
Notion DB의 기존 multi_select 옵션 목록을 프롬프트에 전달. AI가 기존 태그 재활용을 우선하고, 없는 경우에만 신규 생성. Notion은 존재하지 않는 multi_select 값을 자동 생성하므로 별도 처리가 필요 없다.



## 4. URL 없는 인사이트 조사 기능

### 추가된 기능

URL 없이 텍스트만 보내도, Sonnet이 웹 검색으로 해당 주제를 직접 조사하여 Notion에 저장.

**흐름:**

```
텍스트 메시지 (URL 없음) → insight intent 분류
  → Sonnet + 웹 검색 (최대 5회)
  → 조사 결과 → Notion DB 저장
```

### 기술 스택

- Sonnet: 웹 검색 + 리포트 생성
- Lambda timeout: 120s로 증설 (웹 검색 포함 시 60s 초과 발생)

### 문제해결: AI 응답 파싱 방식

Claude가 생성하는 긴 조사 결과를 JSON으로 파싱하려 했으나, 한국어 + 개행 + 특수문자가 포함된 긴 content 필드에서 JSON이 깨지는 파싱 오류가 반복 발생했다.

**결론:**
JSON 포맷을 포기하고 `<title>`, `<content>`, `<sources>` XML 태그 방식으로 교체. `re.search("태그", text, re.DOTALL)`로 파싱하며 긴 텍스트에서도 안정적으로 동작한다.



## 5. 모닝 브리핑

### 추가된 기능

매일 오전 6시 Slack DM으로 자동 발송:

- 날씨 (시간대별 기온 + 강수 예보)
- 오늘 캘린더 일정
- 아스날 / KIA 경기 일정
- F1 세션 일정
- Haiku 한마디 요약 (40자 이내)

### 기술 스택

- Open-Meteo API: 무료, 인증 불필요
- 네이버 스포츠 내부 API: Chrome DevTools 역공학으로 발견
- ESPN API: F1 경기 일정

### 문제해결: 네이버 스포츠 스크래핑 방식

네이버 스포츠는 React SPA라 HTML 스크래핑이 불가능하다. KBO 공식 API는 인증이 필요하고, ESPN에는 KBO 데이터가 없다. Playwright 기반 헤드리스 브라우저는 Lambda에서 실행 비용과 복잡도가 너무 높다.

**결론:**
Chrome DevTools Network 탭으로 네이버 스포츠 앱이 내부적으로 호출하는 `api-gw.sports.naver.com` JSON API를 발견. HTML 스크래핑도 크롤링도 아닌 비공개 API 직접 호출 방식 채택. 인증도 필요 없다.

### 문제해결: 네이버 스포츠 API `selectedDate` 미작동

**인과관계:**
EPL, UCL 등 `selectedDate` 파라미터로 특정 날짜 경기를 조회하려 했으나, 미래 날짜를 전달해도 오늘 데이터만 반환됐다. KBO도 동일하게 `selectedDate`가 무시됐다. 로컬 테스트에서는 오늘 경기가 반환되니 "Lambda가 해당 날짜에 실행되면 정상 작동할 것"이라 판단하고 그냥 뒀다.

이후 사용자가 브라우저에서 다른 날짜 URL이 동작함을 확인하고 DevTools로 내부 API를 직접 확인해 공유. 카테고리마다 파라미터 방식이 완전히 달랐다:

| 카테고리                       | 실제 동작하는 방식                                           |
| :----------------------------- | :----------------------------------------------------------- |
| EPL/UCL/유로파/FA컵/카라바오컵 | `upperCategoryId=wfootball` + `season`으로 시즌 전체 조회 → Python 날짜 필터 |
| KBO                            | `upperCategoryId=kbaseball` + `fromDate=toDate=날짜`         |

**결론:**
모든 카테고리의 `selectedDate` 의존 제거. **교훈: 브라우저 URL의 파라미터와 내부 API 파라미터는 별개. 항상 DevTools로 실제 API 요청을 확인해야 한다.**

### 문제해결: 브리핑 윈도우 범위 설정

**인과관계:**
유럽 축구는 한국 시간 기준 새벽 1~4시에 경기가 열린다. 단순히 "오늘 00:00 ~ 23:59" 범위로 설정하면 새벽 경기가 누락되거나 내일 경기로 분류되는 문제가 발생했다.

**결론:**
브리핑 윈도우를 `당일 00:00 KST ~ 익일 06:00 KST`로 설정. 새벽 경기도 당일 브리핑에 포함되도록 처리.



## 6. Android/Desktop 앱 아키텍처 설계

### 핵심 기술 결정

**UI 프레임워크: Compose Multiplatform (KMP)**

Android Native로 시작하면 나중에 Desktop 앱 추가 시 전면 재작성이 필요하다. 처음부터 KMP 구조로 설계하면 Android + Desktop 코드베이스를 공유할 수 있다. 전원버튼 오버레이, Health Connect 등 Android 전용 기능은 `androidMain`으로 격리하고, 공통 UI는 `commonMain`으로 분리한다.

**이미지 처리 전략 — Sonnet Vision 제거:**

이미지 1장을 Sonnet Vision에 보내면 ~$0.004이나, 온디바이스에서 텍스트로 변환 후 Haiku에 보내면 ~$0.0003으로 최대 92% 비용 절감이 가능하다.

```
ML Kit Image Labeling
  → "screenshot", "document" 판별 → ML Kit OCR → 텍스트만 서버 전송
  → 음식/건물/스포츠 등 → Gemma 4 E4B (온디바이스 VLM) → 설명 텍스트 서버 전송

서버: 항상 텍스트만 수신 → Haiku 처리
Sonnet Vision 비용 = $0
```

### 기술 스택 확정

| 항목            | 결정                          | 이유                                                         |
| :-------------- | :---------------------------- | :----------------------------------------------------------- |
