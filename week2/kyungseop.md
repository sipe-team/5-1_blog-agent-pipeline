## 파이프라인 설계

> 전체 흐름

```mermaid
flowchart LR
    A["수집\n(세션, 커밋,\n노션, 직접입력)"] --> B["시드 처리\n(Sonnet)"]
    B --> C["Blog Seeds DB"]
    C --> N1["Discord 알림\n(시드 N개 추출)"]
    N1 --> D["주제 선택\n(사람)"]
    D --> E["개요 + 초안 작성\n(Sonnet)"]
    E --> F["블로그 DB\n(상태: 초안)"]
    F --> N2["Discord 알림\n(초안 완성)"]
    N2 --> G["검토\n(사람)"]
    G -->|Rewriter| E
    G -->|Publish| H[발행\n(Sonnet)]
    H --> I["kskim.dev PR"]
```

- **수집 소스 (우선순위 순)**:
  1. Claude Code 세션 로그 (일일 작업 맥락이 가장 풍부)
  2. Git commits & diffs (실제 코드 변경 내역)
  3. Notion 메모 (정리해둔 아이디어, 회의록 등)
  4. 사용자 직접 입력 (CLI에서 즉시 시드 등록)
- **사용 툴/프레임워크**: Claude Code (Sonnet), Notion MCP, Discord 웹훅, launchd (nightly 스케줄링)
- **발행 채널**: kskim.dev, PR 머지로 발행

### 단계별 설명

**1단계 — 수집 (Capture)**

두 가지 경로로 데이터가 들어온다.

- **자동 수집 (nightly)**: 매일 23시에 launchd가 스크립트를 실행. Claude Code 세션 로그, 그날의 커밋, Notion 메모를 긁어서 Seed Processor에 넘김. 맥북이 잠자기 상태였으면 깨어났을 때 밀린 작업을 실행해줌.
- **즉시 등록 (수동)**: 개발하다가 '이거 블로그 감이다' 싶으면 `seed-now.sh`로 바로 시드 등록.

**2단계 — 시드 처리 (Seed Processing)**

- Sonnet이 수집된 데이터에서 노이즈 제거 + 핵심 추출 + 주제 분류 + `blog_worthy` 판정까지 한 번에 처리
- 결과는 Notion Blog Seeds DB에 저장
- 처리 완료되면 Discord로 '시드 N개 추출됨' 알림 → Notion 링크 타고 가서 확인

**3단계 — 초안 작성 (Draft)**

- 내가 시드 중에서 쓸 거 골라서 방향 메모를 남기면, Outliner가 개요 잡고 Writer가 초안을 씀 (둘 다 Sonnet)
- 초안은 기존 Notion 블로그 DB에 `상태: 초안`으로 저장 (별도 DB 안 만들고 상태 컬럼 추가)
- 완료되면 Discord로 초안 나왔으니 검토해달라는 식으로 알림

**4단계 — 검토 & 리라이트 (Review)**

- 초안을 읽고 두 가지 중 선택:
  - **직접 편집**: 괜찮으면 손으로 좀 다듬고 바로 발행
  - **리라이트 요청**: '톤이 너무 딱딱하다', '이 부분 더 풀어써라' 같은 피드백만 남기면 Writer가 다시 씀
- 만족할 때까지 반복 가능

**5단계 — 발행 (Publish)**

- 발행 결정하면 Publisher(Sonnet)가 Notion 초안 → MDX 변환 + frontmatter 생성 → kskim.dev 레포에 PR 생성
- PR 머지하면 블로그에 올라가고, Notion에서 `상태: 발행`으로 변경

### 에이전트 구성

| 에이전트 | 트리거 | 역할 | 비용/회 |
|---------|--------|------|---------|
| Seed Processor | nightly cron / 즉시 등록 | 노이즈 제거, 핵심 추출, 주제 분류, blog_worthy 판정 | ~$0.20 |
| Outliner | 시드 선택 시 | 시드 + 방향 메모 → 구조화된 개요 | ~$0.20 |
| Writer | 개요 완성 후 / 리라이트 시 | 개요 → 블로그 초안, 피드백 반영 재작성 | ~$0.30 |
| Publisher | 발행 설정 시 | Notion → MDX 변환 + frontmatter + PR 생성 | ~$0.10 |

전부 Sonnet 단일 모델. 주간 예상 비용 ~$0.8-1.5 (블로그 1편 + 리라이트 1-2회 기준).

### 주요 설계 결정

- **에이전트 통합**: 처음에 Extractor(Haiku) + Structurer(Sonnet) 2단계로 설계했는데, 개인 블로그 수준의 일일 데이터 양이면 Sonnet 하나로 합치는 게 더 단순하고 비용 차이도 거의 없음
- **Writer를 Opus 대신 Sonnet**: Opus는 $2/회로 비싸서, Sonnet($0.30/회)으로 초안 쓰고 리라이트 루프로 품질을 올리는 전략. 사람이 방향만 잡아주면 2-3회 리라이트로 충분할 거라고 판단
- **launchd 선택**: GitHub Actions는 로컬 세션 로그에 접근 못하고, crontab보다 macOS에서는 launchd가 권장됨. 잠자기 복귀 시 밀린 작업도 처리해줌

## 이번 주 진행 내용

- week1에서 만든 이력서 AI 파이프라인(3-agent tmux 병렬)을 실제로 돌려봄
  - WebFetch가 뽑아온 정보가 내 의도와 다른 경우가 많았음
  - tmux 3개 터미널 왔다갔다 하면서 모니터링하는 게 1회성은 괜찮은데 매주 반복하기엔 불편
  - 포트폴리오 같다는 한마디에 전체 방향이 바뀜 — AI 3개 돌리는 것보다 사람 피드백이 결정적
- 위 교훈을 바탕으로 블로그 자동화 파이프라인 설계
  - 핵심: AI한테는 구조화만 시키고, 뭘 쓸지/어떻게 쓸지 판단은 직접 함
  - nightly 자동 수집은 launchd로 매일 23시 실행
  - 직접 아이디어 넣는 즉시 등록 경로도 별도로 열어둠
  - 초안 나오면 직접 고치거나, 피드백만 주고 다시 쓰게 하는 리라이트 루프

## 인사이트 / 배운 것

- 이력서 파이프라인 돌려보니까 AI한테 전부 맡기면 안 된다는 걸 체감했다. WebFetch로 긁어온 정보가 내가 원한 거랑 달라서 결국 다 직접 확인해야 했는데, 그러면 자동화 의미가 없다. 그래서 블로그 파이프라인에서는 AI가 하는 일을 정리까지로 제한하고, 이걸 쓸지 말지에 대한 판단은 내가 하는 구조로 바꿨다.
- 자동 수집만 있으면 안 된다. 개발하다가 이거 블로그에 올리면 좋은 내용이겠다 싶은 순간이 있는데, 그걸 밤 nightly까지 기다리면 맥락을 잊어버린다. 그래서 CLI에서 바로 시드 등록하는 경로를 따로 뒀다.
