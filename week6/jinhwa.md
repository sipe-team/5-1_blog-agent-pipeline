# Week 6 - jinhwa

## 미션의 목표와 진행 과정

이 파이프라인은 한 주 동안 쌓인 Claude/Codex 세션 기록과 메모를 자동으로 정리해, 기술 블로그 초안까지 만들어내는 흐름을 구축하는 것을 목표로 시작했다. 처음에는 세션 로그를 정리하고 초안을 생성하는 자동화에 초점이 있었지만, 진행할수록 단순 요약 자동화보다 "며칠에 걸친 작업 맥락과 문제 해결의 서사까지 어떻게 보존할 것인가"가 더 중요한 과제가 되었다. 최종적으로는 로그 기반 초안 생성뿐 아니라, 사람이 맥락을 보강하고 피드백을 반영해 다시 재생성할 수 있는 구조까지 포함하는 방향으로 발전했다.

### 목표의 변화

처음의 목표는 세션 기록과 메모를 자동으로 정리해 블로그 초안까지 만들어내는 것이었다. 하지만 진행하면서 멀티데이 작업의 맥락과 문제 해결의 서사를 함께 보존하는 일이 더 큰 과제가 되었고, 그에 따라 완전 자동 생성보다는 사람의 리뷰와 재생성을 포함한 구조로 방향이 확장되었다.

### 자동화 실험의 확장성

이 과정을 진행하면서 좋았던 점은, 블로그 글 자동화라는 비교적 개인적인 문제를 다뤄 보면서 평소 해보고 싶었던 여러 자동화 아이디어를 실제로 떠올리고 확장해볼 수 있었다는 점이다. 한 번 파이프라인을 끝까지 만들어보니, 비슷한 방식으로 유튜브 영상 생성 자동화나 개인 일정을 정리하고 관리해주는 자동화도 시도해보고 싶다는 생각이 들었다. 단순히 하나의 결과물을 만든 것보다, 앞으로 어떤 개인용 에이전트나 자동화를 더 만들 수 있을지 감이 생겼다는 점이 의미 있었다.

### 진행 중 드러난 어려움

진행 과정에서는 몇 가지 어려움이 분명하게 드러났다. 초기에 JSONL 로그를 그대로 다루면 ANSI 코드, progress bar, 단순 파일 목록 같은 노이즈가 많아 정제 단계가 필수였고, 모델이 실제로 하지 않은 내용을 보태거나 사용자의 시행착오를 AI의 시점으로 잘못 서술하는 문제도 있었다. 이를 해결하기 위해 Haiku로 노이즈 제거, Sonnet으로 구조화, Gemini로 블로그 초안 작성이라는 다단계 파이프라인을 설계하고 프롬프트를 계속 조정했다. 이후 자동화를 붙이는 과정에서는 macOS의 cron/launchd 환경에서 Claude CLI 인증 토큰이 정상적으로 로드되지 않는 문제가 있었고, 이를 비대화형 환경에 직접 주입하는 방식으로 해결했다.

### 예상과 달랐던 점

처음에는 인풋이 충분히 많고 세션 기록이 풍부하면 결과물도 자연스럽게 잘 나올 것이라고 생각했지만, 실제로는 그렇지 않았다. 로그가 많이 남아 있어도 큰 맥락과 글의 틀을 일관되게 유지하면서 완결성 있는 글을 만들어내는 일은 생각보다 훨씬 어려웠다. 특히 실제 업무는 며칠에 걸쳐 이어지는 경우가 많아 하루치 로그만으로는 맥락이 끊겼고, 그래서 세션을 날짜가 아니라 프로젝트 단위로 묶고 `/blog-tag` 같은 명시적 태깅을 통해 하나의 작업 스레드로 누적 관리하는 방식으로 구조를 바꾸게 되었다.

또 한 가지 큰 인사이트는, 세션 기록만으로는 "무엇을 했는가"는 남아도 "왜 그 선택을 했는가", "어떤 고민 끝에 방향을 정했는가" 같은 맥락은 충분히 남지 않는다는 점이었다. 이 때문에 좋은 블로그 글을 만들려면 생성 파이프라인 자체뿐 아니라, 초안 이후 사람이 맥락을 보강하고 재생성하는 리뷰 루프가 꼭 필요하다는 결론에 도달했다. 결국 이번 작업을 통해 느낀 것은, 완전 자동화는 아직 어렵고, 당분간은 사람이 중간중간 개입해 방향과 맥락을 잡아주는 자동화 정도가 더 현실적이라는 점이었다. 개인적으로도 그 정도 수준만 되어도 충분히 유용하다고 느꼈다.

## 미션 결과물의 구조

파이프라인은 Python 스크립트 기반으로 구현되었고, 입력 소스로는 Claude/Codex 세션 JSONL과 Obsidian의 로컬 Markdown 메모를 사용한다. LLM 단계는 Anthropic 계열 모델인 Haiku와 Sonnet, 그리고 Gemini를 조합해 사용했다. 자동 실행은 macOS의 cron 또는 launchd 기반 나이틀리 잡으로 설계되었고, 세션 수집 후 1차로 Haiku가 로그 노이즈를 제거하고 `blog_worthy` 여부를 판단한다. 가치가 있다고 판단된 세션만 Sonnet이 구조화된 Markdown으로 정리하고, 이후 블로그 초안 생성 단계로 이어지는 구조다.

### 기본 기술 스택

전체 오케스트레이션은 Python 스크립트로 구성되었고, 입력 소스로는 Claude/Codex 세션 JSONL과 Obsidian Markdown 메모를 사용했다. LLM은 Haiku, Sonnet, Gemini를 조합해 사용했고, 실행 방식은 macOS의 cron 또는 launchd 기반 나이틀리 잡으로 설계되었다.

### 멀티데이 컨텍스트를 위한 조치

파이프라인은 처음의 하루 단위 처리 구조에서, 프로젝트 단위 처리 구조로 개편되었다. 이를 위해 `/blog-tag <project>` 슬래시 커맨드와 `UserPromptSubmit` 훅을 이용해 세션을 하나의 작업 스레드로 묶었고, `thread.md`를 누적 재합성해 멀티데이 작업의 맥락을 유지하도록 했다. `--close` 옵션이 붙으면 누적 세션 전체를 바탕으로 최종 글을 생성하는 방식까지 포함된다.

### 후반의 운영 구조 완전 개편

후반부에는 결과물을 파일과 PR 중심으로 다루는 대신 DB 기반 운영 구조로 전환했다. `Postgres/Supabase + Drizzle` 위에 `article`, `article_version`, `review_comment`, `regeneration_job` 중심의 스키마를 두었고, 웹앱은 draft 관리와 리뷰 UI, 상태 관리를 맡고 파이프라인은 internal API를 통해 초안 생성과 regenerate job 처리를 수행하도록 역할을 분리했다.
